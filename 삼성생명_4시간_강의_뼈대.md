각 교시는 독립 주제처럼 보이지만, 하나의 이야기로 이어져야 한다.

```text
1교시  언어   — "에이전트"라는 말을 차분히 풀어보면 남는 것은 Skill이다
2교시  엔진   — 그 Skill을 실행할 모델은 어떻게 고르는가

4교시  손     — 직접 Skill을 만들고 Connector를 연결해본다
```

관통 메시지 (마지막 슬라이드에서 회수):

1. 에이전트는 여러분이 만드는 게 아니라 **이미 존재한다** (Claude Desktop, Claude Code가 에이전트다).

4. 배워야 할 것은 구현 기술이 아니라 **내 도메인 지식을 외부화하는 능력**이다.

## 2. 1교시 (13:30–14:20) — Agent, MCP&Connector, Plugin, Skill: 어떻게 활용하면 좋은가

### 목표 문장
> "참가자가 '에이전트 만들어야 한다던데요?'라는 말을 들었을 때, 그 말이 실제로 가리키는 것을 차분히 구분해 되물을 수 있게 된다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 13:30–13:35 | 훅 | 인트로 시연 + "three.js 모릅니다" + 오늘의 지도 |
| 13:35–13:43 | 스토리 | ChatGPT의 첫날 — 수조 원이 들었던 "웃음거리" |

### 토킹포인트 (슬라이드 순서대로)

**S1-1. ChatGPT의 첫날 (스토리)**
- 2022년 말 ChatGPT가 나왔을 때를 기억하는가. "이게 무슨 AI냐"고 비웃음을 받았다.
- 그 비웃음 받던 물건이 **수조 원**을 들여 만든 것이었다.
- 질문 던지기(중립 화법): "이 어려운 일을 — 어디까지 직접 만들고, 어디서부터 가져다 쓰면 될까요?" — make-or-buy 질문으로. 답을 주지 말고 세션 내내 끌고 간다. (구버전의 "수조 원 vs 몇천만 원" 대비 화법은 조롱조라 폐기)

**S1-2. 에이전트의 진짜 정의**
- 에이전트 = 목표(goal)를 향해 **관찰 → 사고 → 재행동**을 반복하는 것.
- 비유: 신입사원에게 "경쟁사 조사해 와"라고 시키면 → 검색하고(행동), 결과를 보고(관찰), 부족하다 판단하고(사고), 다시 찾는다(재행동). 이 루프가 에이전트다.
- 이를 구현하는 기법들이 있다 — ReAct, Plan & Execute, 서브태스크 컨텍스트 격리(본 작업 오염 방지). 비개발자에겐 이름보다 "즉흥형 / 계획형 / 분업형 일하기 방식"이라고 풀어준다. 슬라이드에 영문 용어는 각주 크기로만.
- 핵심 반전: **에이전트는 여러분이 만드는 게 아니라 이미 있다.** Claude Desktop, Claude Code — 이것들이 에이전트다. 여러분 노트북에 이미 설치할 수 있다.

**S1-3. 모델만큼 중요한 것: 에이전트 설계 (오케스트레이션)**
- 에이전트의 품질 = 모델 × 설계. 설계란: 프롬프트를 얼마나 넣고 얼마나 **덜어낼지**, 관찰을 몇 번 시킬지, 재행동을 몇 번 허용할지, 모델을 얼마나 믿고 맡길지 — 이것들이 유기적으로 맞물린 것.
- 증거: Claude Code에 Gemini 모델을 끼우거나, Gemini CLI에 Claude 모델을 끼우면 **원래 성능이 안 나온다.** 같은 엔진도 차체가 다르면 그 차가 아니다.
- 비유: 유능한 직원(모델)도 팀장이 일을 어떻게 나눠주고 언제 중간보고를 받느냐(설계)에 따라 성과가 갈린다.
- (이 슬라이드가 2교시 "모델 선별"의 복선이 된다.)

**S1-4. "1인 1에이전트"와 노코드 빌더 열어보기**
- 왜 이 말이 퍼졌나: 극적이고, 영업할 때 납득되는 단어라서. 단어의 힘이지 기술의 실체가 아니다.
- 노코드 에이전트 빌더에서 여러분이 실제로 하는 일을 뜯어보면: 지시문을 쓰고(=프롬프트), 어떤 도구를 쓸지 고른다(=도구 선택). **그건 에이전트를 만드는 게 아니라 Skill을 쓰는 것이다.**
- 그럼 관찰·재행동·오류 시 폴백은 누가 하나? → **플랫폼의 오케스트레이터**가 한다. 여러분이 만든 게 아니다. 그 플랫폼을 떠나는 순간 "내 에이전트"는 없다.

**S1-4a. 첫 번째 존재론적 오해 — Agent의 개수는 지능의 개수가 아니다**
- 이 블록이 1교시의 새 중심 명제다: **"1인 1Agent는 대체로 1인 1두뇌가 아니라 1인 1업무 프로필이다."**
- 직원 100명이 빌더에서 Agent를 하나씩 만들어도 독립 AI 모델 100개가 생기는 것이 아니다. 일반적으로는 같은 공용 모델과 실행 인프라가 사람별 역할 카드·기억·권한을 바꿔 읽는다.
- 화면에서 개인별인 것: 이름, 말투, 업무 지침, 저장된 기억, Connector 권한, 진입 화면.
- 대체로 공유하는 것: 기반 모델, 추론 방식, 도구 호출 루프, 재시도·오류 처리, 서버·연산 인프라.
- 강의 화법: **"화면에는 직원 셋이 있지만, 안에서는 같은 범용 연기자가 비서·엔지니어·마케터 유니폼을 갈아입는 구조일 수 있습니다."**
- 중요한 질문: **"여기서 1은 무엇을 세는 숫자입니까? 인터페이스 1개입니까, 설정 1개입니까, 독립 실행기 1개입니까, 별도 지능 1개입니까?"** 보통 앞의 두 개를 세면서 뒤의 두 개처럼 들리게 된다.

**S1-4b. 두 번째 존재론적 오해 — 페르소나는 직함이지 직능이 아니다**
- `"당신은 20년 경력의 수석 엔지니어입니다"`라는 문장이 20년의 경험을 생성하지 않는다. 엔지니어처럼 말하게 할 수는 있지만 실제 엔지니어 업무 수행을 보장하지 않는다.
- 비서라는 역할의 재료: 캘린더·이메일 접근, 우선순위와 선호 시간, 충돌 처리 기준, 외부 발송 승인 규칙, 기억, 에스컬레이션 조건.
- 엔지니어라는 역할의 재료: 실제 저장소, 요구사항, 시스템 제약, 코딩 규칙, 실행 환경, 테스트, 보안 검사, 수정 권한, 실패 시 중단 조건.
- 역할은 명사가 아니라 검증 가능한 동사의 묶음이다: **읽는다 → 판단한다 → 행동한다 → 검사한다 → 실패하면 재시도하거나 사람에게 올린다.**
- 핵심 문장: **"페르소나는 직함을 만들고, 업무 재료가 일을 만듭니다."** 그 재료와 실행 계약을 문서화한 핵심이 Skill이다.

**S1-4c. 세 번째 존재론적 오해 — Agent가 많아도 공통 한계는 반복된다**
- 비서·엔지니어·마케터 Agent가 같은 기반 모델을 사용하면 이름과 컨텍스트는 달라도 같은 지식 공백, 편향, 과신, 그럴듯하게 채우는 경향을 공유할 수 있다.
- 마케터가 쓰고 전략가가 검토하고 CEO가 승인해도 셋이 같은 모델이라면 **같은 사람이 모자를 바꿔 쓰고 자기 문서를 세 번 검토하는 것**일 수 있다.
- 정직한 양보: 역할 분리는 유효하다. 컨텍스트를 격리하고, 서로 다른 자료와 평가기준을 주고, 독립 초안을 만들게 하면 결과가 개선될 수 있다.
- 단, 독립성은 이름이나 프로필 사진에서 생기지 않는다. **다른 근거 자료, 분리된 컨텍스트, 상이한 평가기준, 결정론적 테스트, 사람 검토**에서 생긴다.
- 따라서 `멀티에이전트 = 독립 전문가 N명`이라는 등식은 금지한다. 정확한 표현은 `같은 범용 실행기를 서로 다른 작업 조건에서 N번 운용`이다.

**S1-4d. 1인 회사의 정확한 구조**
- 환상: 사람 1명 + AI 직원 N명.
- 실제에 가까운 구조: **사람 1명 + 공용 AI 실행기 1개 + 역할별 업무 시스템 N개.**
- 업무 시스템의 구성: `Skill + 데이터 + 도구 + 테스트 + 승인 기준`.
- 사람에게 남는 역할: 목표, 권한, 예외 판단, 최종 책임. Agent에게 이름을 붙여도 책임 주체가 새로 생기는 것은 아니다.
- 강의 문장: **"AI 직원 N명을 창조한 것이 아니라, 하나의 범용 AI 실행기를 N개의 검증 가능한 업무 시스템에 반복 배치한 것입니다."**
- 행동 지침: **"모자를 여러 개 만들지 말고, 일을 여러 개 설계하십시오."** Agent 이름보다 입력·판단 기준·도구·산출물·검사·승인 지점을 먼저 적는다.

**S1-4e. 금융사 도입에서 이어지는 질문**
- 온프렘 도입의 함정 (금융사라 반드시 짚기):
  - 빌더들은 "모든 모델 호환"을 홍보하지만, 사내에 들어오는 그 모델로 안정화·프롬프트 최적화·오케스트레이션 이중화가 되어 있을 가능성은 매우 낮다.
  - 그런데 내부는 블랙박스라 확인할 수 없다.
  - SaaS 시연 때는 **파워 좋은 프론티어 모델 + 거기에 맞춘 오케스트레이션**으로 보여주니, 시연과 도입 후 성능이 다를 수밖에 없는 구조다. 속는 게 아니라 속을 수밖에 없다.
- 청중용 질문 리스트 슬라이드 (실무자가 벤더에게 던질 3가지): ① 시연에 쓴 모델과 우리가 도입할 모델이 같은가? ② 우리 모델 기준으로 프롬프트/오케스트레이션 최적화가 됐다는 근거는? ③ 이 플랫폼을 떠나면 우리가 만든 것 중 무엇이 남는가?

**S1-5. 그래서 Skill — 지속가능한 AI 활용 기술 (강의 전체의 핵심 슬라이드군, 시간 최다 배정)**

이 대목이 4시간 전체의 명제다. 슬라이드 3~4장으로 쪼개서 충분히 다룬다.

*(a) 세 번째 질문의 답*
- S1-4의 벤더 질문 ③("이 플랫폼을 떠나면 무엇이 남는가")의 답: 플랫폼을 떠나도 남는 것 = **어디서든 이식 가능한 md(문서) 형태의 Skill.**
- Skill = 내 업무 노하우를 AI가 읽을 수 있는 표준업무절차서로 쓴 것. 파일이니까 내 것이고, 옮길 수 있고, 버전 관리되고, 후임에게 물려줄 수 있다.

*(b) Skill의 실체 — 특별한 기술이 아니라는 것 자체가 설계다*
- 실체는 `SKILL.md`라는 문서 파일 하나 (+ 필요하면 참고자료 파일 몇 개를 같은 폴더에).
- 문서 맨 위에 두 줄: 이 지침의 **이름**과 **"언제 꺼내 읽어야 하는지"** 한 줄 설명. 그 아래에 업무 절차 본문.
- 작동 방식이 사람과 똑같다: AI는 평소엔 맨 위 한 줄 설명만 기억하고 있다가, 해당 업무가 들어오면 그때 본문을 펼쳐 읽는다. 회사 규정집을 통째로 외우는 직원은 없다 — 필요할 때 해당 페이지를 펴는 직원이 유능한 것이다. (기술 용어로는 progressive disclosure — 슬라이드엔 각주로만.)
- 그래서 토큰(=AI의 주의력이자 비용)을 아낀다. 이 단순한 구조가 아래 (c)의 논쟁에서 이긴 이유다.

*(c) 역사: 조롱 → 표준까지 5개월 (팩트체크 완료, 출처는 강사노트)*
- **2025-10-16** Anthropic이 Agent Skills 발표 (claude.ai + Claude Code 동시).
- **발표 직후, 반응은 갈렸다.**
  - 냉소 진영 (Hacker News 실제 코멘트): *"이건 설계 패턴/프롬프트 엔지니어링 트릭이지 표준이 필요한 물건이 아니다"*, *"근본적으로 프롬프트에 텍스트 이어붙여 주는 프레임워크에 왜들 흥분하나?"* — "그냥 마크다운 파일 아니냐"는 반응.
  - 반대편 극단: 업계에서 가장 영향력 있는 개발자 블로거 사이먼 윌리슨(Simon Willison)이 발표 당일 쓴 글 제목 — **"Claude Skills are awesome, maybe a bigger deal than MCP"** (MCP보다 더 큰 물건일지도).
- **2025-12-18** Anthropic이 Skill 규격을 **오픈 표준으로 공개**. 그리고 **48시간 안에** Microsoft가 VS Code에, OpenAI가 ChatGPT와 Codex에 채택했다. OpenAI의 Codex는 지금 Anthropic이 만든 그 `SKILL.md` 형식을 그대로 읽는다 — 최대 경쟁사가 경쟁사의 형식을 수용한 것.
- **2026-03 기준** Google Gemini CLI, GitHub Copilot, JetBrains, AWS 등 **32개 이상의 도구**가 같은 SKILL.md 파일을 같은 폴더 구조에서 읽는다. 사실상의 산업 표준.
- 왜 5개월 만에 표준이 됐나: **그냥 파일이기 때문이다.** 경쟁사가 채택하는 데 SDK도 서버도 계약도 필요 없었다. "그냥 마크다운"이라는 조롱이 가리킨 바로 그 지점이 승리의 이유였다.
- S1-1과 수미상응시킬 것: **ChatGPT도 "이게 무슨 AI냐"고 조롱받았고, Skill도 "이게 무슨 기술이냐"고 조롱받았다. 단순해 보인다는 조롱은 종종 표준이 되기 직전의 신호다.**

*(d) 이것이 왜 "여러분의" 자산인가 (노코드 빌더와의 최종 대비)*

## 3. 2교시 (14:30–15:20) — 프론티어 모델 선별 방법

### 목표 문장
> "참가자가 벤치마크 표 대신, 자기 업무 문제 3개로 모델을 비교·선택하는 절차를 갖게 된다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 14:30–14:38 | 문제 제기 | 벤치마크 1등 모델이 내 업무에서도 1등인가 |
| 14:38–14:48 | 프레임 | 하네스 — 모델을 감싸는 보이지 않는 절반 (1교시 복선 회수) |
| 14:48–15:03 | 지도 | 프론티어 모델 지형도: 성격 + DeepSWE 실측 지표 읽는 법 |
| 15:03–15:08 | 데모 | 리더보드 라이브 리딩 (점수 정렬 vs 비용 정렬) |
| 15:08–15:15 | 확장 | 로컬 모델과 개인 레버리지의 시대 |
| 15:15–15:20 | 절차+요약 | 내 업무 3문제 테스트법 + 한 줄 요약 |

### 토킹포인트

**S2-1. 벤치마크의 함정**
- 모델 발표마다 나오는 점수표 — 그런데 그 시험 문제는 여러분의 업무가 아니다.
- 벤치마크는 "수능 점수", 여러분에게 필요한 건 "우리 팀 실무 면접". 수능 만점자가 우리 팀 보고서를 제일 잘 쓴다는 보장은 없다.

**S2-2. 하네스 — 모델을 감싸는 보이지 않는 절반 (1교시 복선 회수, 상세히)**

- 용어부터: 하네스(harness)는 원래 **말에 채우는 마구(馬具)**다. 아무리 좋은 말도 마구 없이는 수레를 못 끈다. AI에서 하네스 = **모델을 실제 업무에 매어주는 장비 일체.**
- 하네스가 실제로 하는 일 6가지 (슬라이드 1장, 아이콘으로):
  1. **업무 지침** — 모델에게 미리 주는 기본 지시문(말투, 안전 규칙, 일하는 원칙). 여러분이 채팅창에 쓰는 글 이전에 이미 수천 단어가 들어가 있다.
  2. **도구 상자** — 파일 읽기/쓰기, 검색, 프로그램 실행… 무엇을 쥐여줄지, 각 도구의 사용법을 어떻게 설명해둘지.
  3. **권한 체계** — 무엇은 그냥 하게 하고, 무엇은 사용자에게 물어보게 할지.
  4. **컨텍스트 관리** — 대화가 길어지면 요약해서 이어가기. 사람으로 치면 "회의가 길어지면 중간 정리를 해주는 서기".
  5. **반복 제어** — 실패하면 몇 번까지 재시도, 언제 멈추고 보고할지.
  6. **분업** — 큰 일을 쪼개 보조 일꾼(서브에이전트)에게 위임하고 결과만 회수할지.
- 회사 비유로 묶기: 유능한 경력직(모델)을 뽑아도 **온보딩 문서, 시스템 접근 권한, 결재선, 보고 체계**가 없으면 성과가 안 나온다. 하네스는 그 직원에게 회사가 깔아주는 **업무 환경 일체**다.
- 1교시 복선 회수: Claude Code + Gemini 모델 = 성능 하락. 이유가 이제 보인다 — 하네스의 6요소가 전부 **특정 모델의 성격**(어디까지 믿고 맡길지, 지시를 얼마나 주입/절제할지, 몇 번 관찰시킬지)에 맞춰 조율돼 있기 때문. 남의 회사 온보딩 문서로 우리 신입을 교육하는 격.
- 증거 (S2-3 벤치마크의 복선): 모델 비교 벤치마크들은 **일부러 하네스를 가장 단순한 공통 사양으로 고정**하고 잰다(뒤에 볼 DeepSWE는 mini-swe-agent라는 표준 하네스로 통일). 왜? **하네스가 성능을 좌우한다는 걸 벤치마크 제작자들이 알기 때문이다.** 거꾸로 말하면, 여러분이 쓰는 제품의 성능은 이 표와 다르다 — 각 사 제품은 자기 모델에 최적화된 하네스를 얹으니까.
- 3층 구조 그림 (이 슬라이드가 강의 전체의 지도가 된다):

| 층 | 무엇 | 누가 만드나 |
|---|---|---|
| Skill | 내 업무의 절차와 기준 | **여러분 (4교시)** |
| 하네스 | 모델을 업무에 매는 실행 환경 | 제조사 (Claude Desktop/Code = Anthropic의 하네스) |
| 모델 | 지능 그 자체 | 제조사 |

- 실무 시사점: "어느 모델이 최고냐"는 반쪽짜리 질문. **"어느 제품(모델+하네스 묶음)이 내 업무에 맞냐"**가 온전한 질문. Claude Desktop을 쓴다는 건 Claude 모델 + Anthropic의 하네스 묶음을 쓰는 것이다. 그리고 그 묶음 위에서 여러분이 손대는 층은 Skill 하나다.

**S2-3. 프론티어 모델 지형도 (성격으로 소개 + 실측 지표)**
- 스펙 나열 금지. "사람 소개하듯" 소개한다. 축 두 개면 충분: ① 깊이 생각 vs 빠른 처리, ② 대화형 vs 위임형(에이전틱).
- 성격 구도 (강의 시점 최신 기준으로 전날 업데이트할 것):
  - Claude Fable 5 (Claude 5 패밀리) — 가장 깊게 생각하는 최상급 시니어. 복잡한 분석·긴 위임 업무. 대신 비싸다.
  - Claude Opus 4.8 / Sonnet 5 — 깊이와 균형의 중간층. 일상 업무의 기본값.
  - Claude Haiku 4.5 — 빠르고 싼 대량 처리. 분류, 추출, 반복 작업.
  - GPT 계열 / Gemini 계열 — 각자의 강점 영역과 생태계(오피스 연동, 검색 연동 등) 관점으로 공정하게.
- **실측 지표로 성격 보여주기 — DeepSWE 리더보드** (datacurve.ai의 오염 없는 실전형 벤치마크: 실제 저장소 91개에서 새로 만든 113개 과제, 전 모델을 mini-swe-agent라는 **동일 표준 하네스**로 측정 → S2-2 복선 회수). v1.1 발췌 (2026-07-17 확인, 강의 전날 재확인):

| 모델 [thinking 강도] | 성공률(Pass@1) | 평균 비용 | 출력 토큰 | 스텝 수 |
|---|---|---|---|---|
| gpt-5.6-sol [max] | 73%±3% | $8.39 | 60k | 61 |
| claude-fable-5 [max] | 70%±4% | $21.63 | 119k | 88 |
| gpt-5.6-terra [max] | 70%±3% | $4.95 | 72k | 76 |
| gpt-5.5 [xhigh] | 67%±6% | $7.23 | 46k | 82 |
| claude-opus-4.8 [max] | 59%±2% | $13.22 | 135k | 120 |
| claude-sonnet-5 [max] | 54%±4% | $26.40 | 214k | 268 |
| gemini-3.5-flash [medium] | 37%±2% | $7.34 | 276k | 86 |
| claude-sonnet-4.6 [high] | 30%±4% | $5.52 | 76k | 134 |
| gemini-3.1-pro [high] | 12%±2% | $9.48 | 196k | 81 |

- 이 표를 "성격"으로 읽는 법 (점수 열이 아니라 나머지 열을 보게 하라):
  - **같은 70%인데 일하는 방식이 다르다**: claude-fable-5는 119k 토큰·88스텝·$21.63을 쓰며 깊게 파고, gpt-5.6-terra는 72k·76스텝·$4.95로 효율적으로 도달. "누가 더 낫냐"가 아니라 "심사숙고형 vs 속전속결형" — 사람 성격과 같다.
  - **claude-sonnet-5의 268스텝**: 한 방의 깊이 대신 많이 두드려서 따라가는 스타일. 그래서 토큰이 214k로 가장 많고 비용도 역전된다.
  - **gemini-3.1-pro의 12%**: 모델이 나쁘다는 뜻이 아니다. 이 표준 하네스와의 궁합, 그리고 측정된 thinking 강도가 [high]로 달랐다는 조건을 봐야 한다. → "표 하나로 모델을 사형선고하지 마라"는 교훈 그 자체.
- **[max]/[xhigh]/[high]가 뭔가 — thinking 강도 다이얼**:
  - 요즘 프론티어 모델엔 "얼마나 오래 생각하고 답할지"를 조절하는 다이얼이 있다 (Claude는 effort, GPT는 reasoning effort라고 부른다). 리더보드의 대괄호가 그 설정값.
  - 정직한 주의: 같은 모델을 max/xhigh/high로 나란히 잰 공개 표는 드물다 (DeepSWE도 모델마다 실린 강도가 다름). 그리고 강도를 올린다고 항상 좋아지는 것도 아니다 — 일부 공개 평가에선 medium→high에서 오히려 점수가 내려가는 수확 체감이 보고됐다. **"다이얼 최대 = 항상 최선"이 아니라 "과제 난이도에 맞는 강도"가 정답.** (비유: 모든 결재 서류를 이사회 안건처럼 검토하면 회사가 멈춘다.)
- **"Max로 가면 적대 에이전트가 내 답을 리뷰한다던데?" — 소문 팩트체크 (Q&A 대비, 슬라이드엔 넣지 않아도 강사는 알고 있을 것)**:
  - 공식 문서상 thinking/effort 다이얼은 "생각에 쓰는 토큰 예산 조절"이다. 다이얼을 올리면 별도 적대 에이전트가 붙는다는 공식 확인은 **없다**. 내부는 비공개(블랙박스)가 맞다.
  - 다만 소문의 뿌리는 실재한다. 다이얼이 아니라 **별도의 최상위 제품군**에서: OpenAI GPT-5 Pro는 "병렬 test-time compute"(여러 추론 경로를 동시에 돌리고 최선을 합성)를 공식 표현으로 쓰고, Google Gemini Deep Think는 여러 내부 에이전트가 병렬로 풀고 상위 레이어가 합성한다고 밝혔으며, xAI Grok Heavy는 병렬 에이전트를 띄워 **하나가 만들면 다른 하나가 비판(critique)** 하는 구조로 알려져 있다 — 소문 속 "적대적 리뷰"에 가장 가까운 공개 사례.
  - 그리고 이 패턴은 하네스 층에서도 공공연히 쓰인다: 예컨대 Claude Code의 코드리뷰 기능은 강도를 올리면 찾아낸 결함을 별도 검증 단계로 재확인시킨다. 즉 "적대적 리뷰"는 모델 내부의 비밀이라기보다 **제품·하네스 설계 패턴**으로 이미 업계 표준이 되어가는 중.
  - 강의용 안전 화법: "다이얼을 올리면 적대 리뷰가 돈다"고 단정하지 말 것. **"돈과 시간을 더 쓰는 최상위 모드들은 여러 답안을 만들어 서로 교차검증시킵니다. 여러분이 중요한 보고서를 두 명에게 각각 쓰게 하고 상호 검토시키는 것과 같은 원리입니다"** 까지가 사실에 부합한다.
- 업무 유형 → 추천 조합 표를 만들되 "정답표"가 아니라 "출발점"이라고 명시.
- 비용 이야기는 비개발자 언어로: "Haiku로 할 일을 Fable에 시키는 건 임원에게 우편물 분류를 시키는 것."

**S2-4. 라이브 데모: 리더보드를 직접 읽어본다**
- (기존의 "같은 프롬프트, 두 모델 비교" 데모는 폐기 — 일상 과제 수준에서는 프론티어 모델 간 차이가 눈에 안 보여 데모가 죽는다. 모델이 실제 일하는 모습은 어차피 3교시 데모에서 보게 된다.)

**S2-5. 내 업무 3문제 테스트법 (가져갈 절차)**

4. 점수 + 속도 + 비용으로 결정하고, **분기마다 재평가한다** (모델은 계속 바뀐다).

**S2-6. 확장: 로컬 모델과 개인 레버리지의 시대**

**S3-2b. 확장의 문법 — Skill · Connector · Plugin (신규 블록, 1·2교시에서 미소개분 회수)**

*(a) 확장 지도 한 장*

  - **Connector·MCP (회청)** — 손이 닿는 범위를 늘린다. 표준 창구.
  - **Plugin (인주)** — 팀으로 퍼뜨린다. 검증한 묶음을 배포.
  - **모델·런타임 (회청)** — 손대지 않는 층. 2교시의 그 층.

*(b) MCP — 연결의 표준*
- 그림: 규격이 없다면 도구 4 × 시스템 5 = **개별 연동 20개** → MCP라는 공통 규격이면 4 + 5 = **표준 연결 9개**.
- Connector = 소비자용 이름, MCP = 그 규격의 이름. 비유는 충전 단자 통일.
- 남길 문장 하나: **"Skill이 '일하는 법'의 표준이라면, MCP는 '연결'의 표준이다. 둘 다 규격이라서 도구를 갈아타도 남는다."** — 용어 암기 요구 금지.

*(c) 창구 3종과 신뢰 경계 (금융사 핵심 슬라이드)*

  2. **원격 MCP** — 공식 커넥터 없는 사내 시스템. 읽기 전용·권한 범위를 IT가 통제, 감사 로그 가능한 구조로. → 우리 서버까지.
  3. **로컬 MCP** — 이 노트북 밖으로 나가지 않는다. 네트워크가 끊겨도 동작. 잠시 후 시연할 "맥 비서". → 이 책상 위까지.
- 붙이기 전 다섯 가지: 누가 만들었나 · 무엇을 읽나 · 무엇을 쓰나 · 기록이 남나 · **어떻게 끊나.**
- 문장: **"금융사에서 신뢰는 붙이는 법이 아니라 끊는 법을 아는 데서 생깁니다."**

*(d) Plugin*
- 재료(SKILL.md 여러 개 + 참고자료·입력서식 + Connector 설정 + 공용 요청 문장) → 한 상자 → 팀 전원이 같은 기준으로.
- **"Plugin은 새로운 능력이 아니라 배포 단위입니다."** 팀 표준 업무 키트.
- 정직하게 말할 것: 경계가 아직 정리 중인 개념이고 오늘 만들 대상은 아니다. 메시지는 하나 — **"Skill 하나는 내 자산, Skill 묶음은 팀 자산."** 4교시에 만드는 파일의 다음 단계 예고편으로만 쓴다.

**S3-3. 라이브 데모 ①: 같은 과제, 두 도구**
- 과제: 샘플 문서 묶음(가상 스타트업 자료 등)으로 비교표 만들기.
- Desktop: 파일을 끌어다 넣고 대화로. → 장점: 즉각적. 한계: 문서가 30개면?
- Code: 폴더를 주고 "전부 읽고 비교표 파일로 저장해줘". → 파일이 **생겨나는** 순간을 보여준다. 이 장면이 비개발자에게 가장 임팩트가 크다.

**S3-3b. 라이브 데모 ②: 맥 비서 — Connector를 실제로 붙인다 (4교시에서 이동)**
- 3단계로 진행: ① **붙인다** — 설정 → Connectors, 로컬 창구 "맥 비서"가 도구 5개와 함께 목록에 나타남 ② **시킨다** — 한 문장으로 네 가지(상품 DB 조회 → 40대 가장 관점 비교 → 결론을 소리 내어 읽기 → 알림·미리알림 등록) ③ **끊는다** — 연결 해제 자리를 같은 화면에서 확인.
- 콤보 프롬프트 원문은 `local_mcp/README.md`. 사전 확인: 스피커 볼륨, 방해금지 모드 해제, Claude Desktop 재시작 후 Connectors 표시.
- 데이터는 전부 가상. 응답에도 "가상 데이터" 표기가 붙는다는 점을 화면에서 짚는다.
- 마무리 멘트: "방금 AI가 사내 DB를 조회하고, 보고하고, 후속 일정까지 잡았습니다. 공식 커넥터가 없는 시스템도 이렇게 창구를 만들 수 있습니다."
- **왜 3교시로 옮겼나**: (c)의 로컬 MCP 설명 직후에 실물을 보여줘야 "자료가 가는 거리"가 몸으로 이해된다. 4교시는 참가자 손이 움직이는 Gmail 공식 커넥터 실습에 집중시킨다.

**S3-4. 인트로 제작기 (떡밥 회수, 이 세션의 하이라이트)**
- Before/After 화면 + 실제 사용한 프롬프트 공개.
- 핵심 고백: "저는 이걸 만들기 위해 three.js(3D 프론트엔드 기술)를 배우지 않았습니다. 제가 배운 것은 **장면, 연출, 카메라 모션, 스크롤 이벤트** — 즉 감독의 언어였습니다."
- 프롬프트의 변화 과정을 보여준다: "멋있게 만들어줘"(실패) → "카메라가 아래에서 위로 도롤리하며, 스크롤에 따라 장면이 전환되고…"(성공). **구현은 AI의 몫, 연출은 나의 몫.**

### 실습 1: 내 업무 Skill 만들기 (25분)

공통 예제로 먼저 시연(5분) → 개인 적용(20분). 공통 예제: **"경영진 보고용 1페이지 메모" Skill** (기존 기획안의 템플릿 재사용).

참가자 진행 단계:
1. 최근 한 달간 2번 이상 반복한 업무 하나를 고른다.
2. 종이(또는 메모장)에 3가지만 적는다: ① 언제 이 일을 하는가 ② 일하는 순서 ③ 결과물이 갖춰야 할 형식.
3. Claude에게 그대로 붙여넣고 요청한다: "이걸 재사용 가능한 Skill(업무 지침서)로 만들어줘. 빠진 정보가 있으면 질문 5개까지만 해줘." (기존 기획안 8장의 데모 프롬프트 재사용)
4. Claude의 질문에 답하며 지침을 완성한다.
5. Claude Desktop에 Skill로 등록한다. ⚠️ 등록 UI 경로는 버전에 따라 바뀌므로 **강의 전날 실제 화면을 캡처해서 슬라이드에 반영** (아래 체크리스트).
6. 샘플 입력으로 실행해보고, 아쉬운 점 1가지를 지침에 추가한다 — "Skill은 한 번에 완성되지 않고 기른다"는 체험이 목적.

성공 기준: 화려함이 아니라 **"내 지침대로 결과 형식이 바뀌는 것"을 눈으로 확인**하는 것.

### 실습 2: Connector 연결 — Gmail로 간단하게 (20분)

방침: **거창하게 가지 않는다.** 공식 Gmail 커넥터 1개면 "출입증을 준다 → AI가 내 시스템을 대신 들여다본다"는 체험은 완성된다.

1. Connector가 무엇인지 10초 복습: "AI에게 시스템 출입증을 주는 것. 단, 내 권한 이상은 절대 못 본다."
2. 설정 → Connectors → **Gmail 공식 커넥터** 연결 → 구글 로그인 승인(OAuth). 개인 계정 기준이라 사내망 계정 이슈가 없고, 공식 커넥터라 가장 안전하다.
3. 연결 후 과제 (자기 메일함이라 체감이 가장 크다): "지난 2주 받은 메일 중 아직 회신 안 한 것 같은 메일을 찾아 3줄씩 요약하고 발신자를 표시해줘."
4. 반드시 함께 가르칠 것: 연결 **해제**하는 법, 권한 확인하는 법. 금융사에서는 "붙이는 법"보다 "끊는 법"을 아는 게 신뢰를 산다.
5. Custom Connector 시연(맥 비서 로컬 MCP)은 **3교시 S3-3b로 이동**했다. 여기서는 "3교시에서 본 세 종류 중 ① 공식 커넥터를 직접 붙여봅니다"로 연결만 하고, 참가자 손이 움직이는 데 시간을 쓴다. (Gmail이 막히면 플랜 B로 맥 비서를 다시 꺼낸다.)

### 플랜 B (반드시 준비)

- **Gmail OAuth 차단 시** (금융사 망분리/프록시 가능성 높음): 강사 노트북의 **로컬 MCP 서버**로 전환 — 미리 만들어 둔 읽기 전용 서버(가상 보험상품/샘플 문서 조회)를 강사 화면에서 연결·조회 시연하고, 참가자는 Skill 고도화에 시간을 더 쓴다. 로컬 서버라 네트워크가 완전히 죽어도 동작한다.

- **전체 마비 시**: 전 과정을 강사 화면 시연 + 참가자는 종이에 Skill 설계서 작성 (2번 단계까지는 오프라인으로 가능).

## 5-2. 용어 점검 결과 (2026-07-21, 외부 출처 검증 완료)

강의에서 쓰는 조어성 표현을 실제 업계 용례와 대조한 결과. **핵심 원칙: 내가 만든 구분은 만든 구분이라고 밝힌다. 밝히면 신뢰가 올라가고, 감추면 검색 한 번에 무너진다.**

| 표현 | 판정 | 근거 / 조치 |
|---|---|---|
| **하네스 (harness)** | ✅ **완전한 업계 공용어** | Google·OpenAI·Microsoft·Anthropic·Salesforce·Databricks·Red Hat이 모두 공식 문서에서 사용. arXiv 논문 다수. 그대로 쓸 것 |
| **런타임 하네스** | ✅ 사실상 표준 정의와 일치 | 연구·업계 정의가 "harness = **runtime substrate**"(Salesforce, arXiv 2605.13357). 강조 표현일 뿐 창작 아님 |
| **업무 하네스** | ⚠️ **강의용 구분 (조어)** | "work/business harness"라는 용어는 업계에 없음. **단, 가리키는 대상은 실재**하고 Red Hat은 오히려 "하네스 = AGENTS.md·skills·프롬프트·테스트"라는 설계층 쪽으로 정의함 → 업계가 한 단어로 두 층을 오가는 상태. 슬라이드 각주로 이 사실을 밝히고 "오늘은 나눠 부르겠습니다"로 처리 (2교시 역할 전환 장표) |
| ~~BUSINESS HARNESS~~ | 🔧 수정함 | 영문 칩이 실재 용어처럼 보여 `WORK DESIGN LAYER`로 변경 (검색해도 안 나오는 영문 조어를 만들지 않기 위해) |
| **Human Gate** | ⚠️ 자명하지만 비표준 | 표준어는 human-in-the-loop. 제공 PDF도 "사람 검토·승인"으로만 표기하고 이 단어는 안 씀. **첫 등장(3교시 퍼널 장표)에 "사람이 승인하는 지점" 병기**하도록 수정함 |
| **연출 언어** | ✅ 문제없음 | 명백한 비유이고 그렇게 소개됨. 비유는 출처가 필요 없다 |
| **즉흥형·계획형·분업형** | ✅ 문제없음 | ReAct / Plan-and-Execute / 서브태스크 격리를 풀어 쓴 것이라고 이미 밝히고 있음 |

**추가된 근거 장표 (2교시, 데크 40번)**: "하네스는 이미 업계가 함께 쓰는 말입니다" — Google 백서 *The New SDLC With Vibe Coding*(Osmani·Saboo·Kartakis)의 **모델 10% : 하네스 90%** 비율 + 같은 모델로 하네스만 바꿔 Terminal Bench 2.0에서 30위권 밖 → 5위권 사례 + LangChain의 +13.7점 사례. 1교시 "같은 엔진, 다른 차체" 주장의 실측 근거이기도 하다. ⚠️ 10:90은 백서의 대략적 표현이지 측정치가 아니라고 반드시 덧붙일 것.
