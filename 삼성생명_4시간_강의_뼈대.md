## 0. 4시간을 관통하는 내러티브

각 교시는 독립 주제처럼 보이지만, 하나의 이야기로 이어져야 한다.

```text
1교시  진실   — "에이전트"라는 말의 거품을 걷어내면 남는 것은 Skill이다
2교시  엔진   — 그 Skill을 실행할 모델은 어떻게 고르는가
3교시  작업대 — 그 모델을 어떤 도구(Desktop/Code)에서 쓰는가
4교시  손     — 직접 Skill을 만들고 Connector를 연결해본다
```

관통 메시지 (마지막 슬라이드에서 회수):

1. 에이전트는 여러분이 만드는 게 아니라 **이미 존재한다** (Claude Desktop, Claude Code가 에이전트다).
2. 여러분이 만들 수 있고, 만들어야 하고, 만들면 남는 것은 **Skill — 이식 가능한 md 문서**다.
3. 모델은 벤치마크가 아니라 **내 업무 3문제**로 고른다.
4. 배워야 할 것은 구현 기술이 아니라 **내 도메인 지식을 외부화하는 능력**이다.

## 2. 1교시 (13:30–14:20) — Agent, MCP&Connector, Plugin, Skill: 어떻게 활용하면 좋은가

### 목표 문장
> "참가자가 '에이전트 만들어야 한다던데요?'라는 말을 들었을 때, 그 말의 실체를 해부해서 되물을 수 있게 된다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 13:30–13:35 | 훅 | 인트로 시연 + "three.js 모릅니다" + 오늘의 지도 |
| 13:35–13:43 | 스토리 | ChatGPT의 첫날 — 조롱받던 수조 원 |
| 13:43–13:53 | 개념 | 에이전트의 진짜 정의: 관찰→사고→재행동 루프 |
| 13:53–14:03 | 해부 | "1인 1에이전트"와 노코드 빌더의 실체 |
| 14:03–14:13 | 대안 | 그래서 Skill — 이식 가능한 md, 그리고 용어 정리 |
| 14:13–14:20 | 데모+요약 | Claude Desktop이 루프 도는 모습 라이브 + 한 줄 요약 |

### 토킹포인트 (슬라이드 순서대로)

**S1-1. ChatGPT의 첫날 (스토리)**
- 2022년 말 ChatGPT가 나왔을 때를 기억하는가. "이게 무슨 AI냐"고 비웃음을 받았다.
- 그 비웃음 받던 물건이 **수조 원**을 들여 만든 것이었다.
- 질문 던지기: "수조 원을 들여도 조롱받던 그 일을, 몇 천만 원짜리 프로젝트로 '우리 회사 전용 에이전트'를 만들면 될까요?" — 답을 주지 말고 세션 내내 끌고 간다.

**S1-2. 에이전트의 진짜 정의**
- 에이전트 = 목표(goal)를 향해 **관찰 → 사고 → 재행동**을 반복하는 것.
- 비유: 신입사원에게 "경쟁사 조사해 와"라고 시키면 → 검색하고(행동), 결과를 보고(관찰), 부족하다 판단하고(사고), 다시 찾는다(재행동). 이 루프가 에이전트다.
- 이를 구현하는 기법들이 있다 — ReAct, Plan & Execute, 서브태스크 컨텍스트 격리(본 작업 오염 방지). 비개발자에겐 이름보다 "즉흥형 / 계획형 / 분업형 일하기 방식"이라고 풀어준다. 슬라이드에 영문 용어는 각주 크기로만.
- 핵심 반전: **에이전트는 여러분이 만드는 게 아니라 이미 있다.** Claude Desktop, Claude Code — 이것들이 에이전트다. 여러분 노트북에 이미 설치할 수 있다.

**S1-3. 모델만큼 중요한 것: 에이전트 설계 (오케스트레이션)**
- 에이전트의 품질 = 모델 × 설계. 설계란: 프롬프트를 얼마나 넣고 얼마나 **덜어낼지**, 관찰을 몇 번 시킬지, 재행동을 몇 번 허용할지, 모델을 얼마나 믿고 맡길지 — 이것들이 유기적으로 맞물린 것.
- 증거: Claude Code에 Gemini 모델을 끼우거나, Gemini CLI에 Claude 모델을 끼우면 **원래 성능이 안 나온다.** 같은 엔진도 차체가 다르면 그 차가 아니다.
- 비유: 유능한 직원(모델)도 팀장이 일을 어떻게 나눠주고 언제 중간보고를 받느냐(설계)에 따라 성과가 갈린다.
- (이 슬라이드가 2교시 "모델 선별"의 복선이 된다.)

**S1-4. "1인 1에이전트"와 노코드 빌더의 해부**
- 왜 이 말이 퍼졌나: 극적이고, 영업할 때 납득되는 단어라서. 단어의 힘이지 기술의 실체가 아니다.
- 노코드 에이전트 빌더에서 여러분이 실제로 하는 일을 뜯어보면: 지시문을 쓰고(=프롬프트), 어떤 도구를 쓸지 고른다(=도구 선택). **그건 에이전트를 만드는 게 아니라 Skill을 쓰는 것이다.**
- 그럼 관찰·재행동·오류 시 폴백은 누가 하나? → **플랫폼의 오케스트레이터**가 한다. 여러분이 만든 게 아니다. 그 플랫폼을 떠나는 순간 "내 에이전트"는 없다.
- 온프렘 도입의 함정 (금융사라 반드시 짚기):
  - 빌더들은 "모든 모델 호환"을 홍보하지만, 사내에 들어오는 그 모델로 안정화·프롬프트 최적화·오케스트레이션 이중화가 되어 있을 가능성은 매우 낮다.
  - 그런데 내부는 블랙박스라 확인할 수 없다.
  - SaaS 시연 때는 **파워 좋은 프론티어 모델 + 거기에 맞춘 오케스트레이션**으로 보여주니, 시연과 도입 후 성능이 다를 수밖에 없는 구조다. 속는 게 아니라 속을 수밖에 없다.
- 청중용 질문 리스트 슬라이드 (실무자가 벤더에게 던질 3가지): ① 시연에 쓴 모델과 우리가 도입할 모델이 같은가? ② 우리 모델 기준으로 프롬프트/오케스트레이션 최적화가 됐다는 근거는? ③ 이 플랫폼을 떠나면 우리가 만든 것 중 무엇이 남는가?

**S1-5. 그래서 Skill — 지속가능한 AI 활용 기술**
- 세 번째 질문의 답: 플랫폼을 떠나도 남는 것 = **어디서든 이식 가능한 md(문서) 형태의 Skill.**
- Skill = 내 업무 노하우를 AI가 읽을 수 있는 표준업무절차서로 쓴 것. 파일이니까 내 것이고, 옮길 수 있고, 버전 관리되고, 후임에게 물려줄 수 있다.
- 역사적 검증: Anthropic이 Skill을 처음 발표했을 때 클로드 커뮤니티에서조차 비판과 냉담한 반응을 받았다. 그러나 결국 OpenAI도, Google도 이 개념을 수용했다. 지금은 Skill / Plugin(아직 경계가 덜 정리된 개념) / Connector(여전히 MCP라고도 불림)로 자리 잡았다.
- 용어 정리 표 (유인물과 동일):

| 용어 | 한 줄 정의 | 회사 비유 | 오늘 다루는 수준 |
|---|---|---|---|
| Skill | AI에게 주는 재사용 가능한 업무 지침서 (md 문서) | 표준업무절차서 | **직접 만든다 (4교시)** |
| Plugin | Skill·설정·연결을 묶어 배포하는 패키지 | 팀 표준 업무 키트 | 개념 + 설치 시연 |
| Connector (MCP) | AI가 외부 시스템에 접근하는 표준 연결 통로 | 시스템 권한 있는 창구 | **직접 연결한다 (4교시)** |
| Agent | 위 재료로 관찰→사고→재행동 루프를 도는 실행 주체 | 일 위임받은 주니어 동료 | 만들지 않는다. 이미 있다 |

**S1-6. 라이브 데모 (5분)**
- Claude Desktop에 실제 업무형 요청 하나 (예: "이 보도자료 3건을 읽고 경영진용 1페이지 비교 메모 작성").
- 화면에서 루프를 **중계**한다: "지금 관찰했죠 → 부족하다고 판단했죠 → 다시 행동하죠. 이게 에이전트입니다. 여러분은 방금 에이전트를 쓰신 겁니다."

**한 줄 요약**: "에이전트는 이미 있다. 내가 만들 것은 에이전트가 아니라, 에이전트에게 줄 일하는 법 — Skill이다."

### 예상 반론과 답변
- "우리 회사가 도입한 에이전트 빌더는 그럼 쓸모없나요?" → "아니요. 좋은 실행 환경입니다. 다만 그 안에서 여러분이 만드는 자산이 무엇인지 정확히 알고, 그 자산(지침=Skill)을 플랫폼 밖에서도 살 수 있는 md 문서로 관리하자는 겁니다."
- "그럼 '1인 1에이전트'는 하지 말라는 건가요?" → "'1인 1에이전트'의 진짜 의미는 '1인 1 업무지침 자산화'입니다. 그건 오늘 4교시에 실제로 하나 만들어 갑니다."

## 3. 2교시 (14:30–15:20) — 프론티어 모델 선별 방법

### 목표 문장
> "참가자가 벤치마크 표 대신, 자기 업무 문제 3개로 모델을 비교·선택하는 절차를 갖게 된다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 14:30–14:38 | 문제 제기 | 벤치마크 1등 모델이 내 업무에서도 1등인가 |
| 14:38–14:48 | 프레임 | 모델 × 하네스(설계) 궁합 — 1교시 복선 회수 |
| 14:48–15:00 | 지도 | 프론티어 모델 지형도: 성격으로 이해하기 |
| 15:00–15:08 | 데모 | 같은 업무 프롬프트, 두 모델 비교 라이브 |
| 15:08–15:15 | 확장 | 로컬 모델과 개인 레버리지의 시대 |
| 15:15–15:20 | 절차+요약 | 내 업무 3문제 테스트법 + 한 줄 요약 |

### 토킹포인트

**S2-1. 벤치마크의 함정**
- 모델 발표마다 나오는 점수표 — 그런데 그 시험 문제는 여러분의 업무가 아니다.
- 벤치마크는 "수능 점수", 여러분에게 필요한 건 "우리 팀 실무 면접". 수능 만점자가 우리 팀 보고서를 제일 잘 쓴다는 보장은 없다.

**S2-2. 모델 × 설계 궁합 (1교시 복선 회수)**
- 1교시에서 본 것: Claude Code + Gemini 모델 = 성능 하락. 이유: 하네스(설계)가 특정 모델의 성격 — 어디까지 믿고 맡길지, 프롬프트를 얼마나 주입/절제할지, 몇 번 관찰시킬지 — 에 맞춰 조율돼 있기 때문.
- 실무 시사점: "어느 모델이 최고냐"는 반쪽짜리 질문. **"어느 제품(모델+설계 묶음)이 내 업무에 맞냐"**가 온전한 질문. Claude Desktop을 쓴다는 건 Claude 모델 + Anthropic의 오케스트레이션 묶음을 쓰는 것이다.

**S2-3. 프론티어 모델 지형도 (성격으로 소개)**
- 스펙 나열 금지. "사람 소개하듯" 소개한다. 축 두 개면 충분: ① 깊이 생각 vs 빠른 처리, ② 대화형 vs 위임형(에이전틱).
- 예시 구도 (강의 시점 최신 기준으로 전날 업데이트할 것):
  - Claude Opus 4.8 — 깊게 생각하는 시니어. 복잡한 분석, 긴 문서, 위임형 업무.
  - Claude Sonnet 5 — 속도와 품질의 균형. 일상 업무의 기본값.
  - Claude Haiku 4.5 — 빠르고 싼 대량 처리. 분류, 추출, 반복 작업.
  - GPT 계열 / Gemini 계열 — 각자의 강점 영역과 생태계(오피스 연동, 검색 연동 등) 관점으로 공정하게.
- 업무 유형 → 추천 조합 표를 만들되 "정답표"가 아니라 "출발점"이라고 명시.
- 비용 이야기는 비개발자 언어로: "Haiku로 할 일을 Opus에 시키는 건 임원에게 우편물 분류를 시키는 것."

**S2-4. 라이브 데모: 같은 프롬프트, 두 모델**
- 청중 업무와 가까운 과제 하나 (예: "이 보험 상품 요약서에서 고객 오해 소지가 있는 문장을 찾아라").
- 두 모델의 답을 나란히 놓고 청중에게 "어느 쪽을 결재하시겠습니까?" 손들게 한다. → "여러분 방금 모델 선별하셨습니다. 이게 벤치마크보다 정확합니다."

**S2-5. 내 업무 3문제 테스트법 (가져갈 절차)**
1. 내 업무에서 자주 하는 과제 3개를 고른다 (요약 1, 판단 1, 작성 1).
2. 각 과제에 "좋은 결과의 기준"을 먼저 적는다 (모델 보기 전에!).
3. 후보 모델들에 동일하게 시키고, 기준표로 채점한다.
4. 점수 + 속도 + 비용으로 결정하고, **분기마다 재평가한다** (모델은 계속 바뀐다).

**S2-6. 확장: 로컬 모델과 개인 레버리지의 시대**
- 이제 프론티어 모델만의 세상이 아니다. 로컬 모델도 크게 발전했다.
- 사례 스케치 (쇼츠 공장): Mac Studio 한 대 → 로컬 text-to-video 모델 탑재 → 커뮤니티 인기글 자동 수집(배치 크롤링) → Claude/GPT가 기획·프롬프트 작성과 로컬 모델 **감독** → 쇼츠/영상 공장. 개인이 만드는 미디어 생산 라인.
- 안드레이 카파시(전 OpenAI/Tesla) 인용: 1인이 조 단위 가치를 만들 수 있는 시대가 왔다는 관측. ⚠️ 원문 표현 확인 후 인용.
- 스팀 게임 사례: 2인 개발팀이 만든 게임이 이미 수백억 원대 수익. ⚠️ 게임명("메차 카멜리온")·수익(690억+)·개발 인원은 강의 전 반드시 재확인 — 아래 검증 항목 참고.
- 이 이야기의 역할: 겁주기가 아니라 **"레버리지가 개인에게 왔다. 그 레버리지의 손잡이가 오늘 배우는 Skill이다"**로 3교시 연결.

**한 줄 요약**: "모델은 점수표가 아니라 내 업무 3문제로 고른다. 그리고 모델은 항상 '설계와의 묶음'으로 고른다."

## 4. 3교시 (15:30–16:20) — Claude Code와 Claude Desktop: 차이와 상황별 선택 기준

### 목표 문장
> "참가자가 월요일 아침에 두 도구 중 무엇을 열지 3초 안에 판단할 수 있게 된다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 15:30–15:40 | 구조 | 두 도구의 본질적 차이 |
| 15:40–15:50 | 기준 | 상황별 선택 매트릭스 + 3초 판별법 |
| 15:50–16:00 | 데모 | 같은 과제를 두 도구로 라이브 |
| 16:00–16:15 | 제작기 | 인트로 화면은 어떻게 만들어졌나 (떡밥 회수) |
| 16:15–16:20 | 요약 | 배움의 전환 + 한 줄 요약 |

### 토킹포인트

**S3-1. 본질적 차이 한 장**
- Claude Desktop = **대화 상대**. 내가 자료를 가져가서 보여주며 함께 일한다. 화면 중심, 왕복 중심.
- Claude Code = **내 컴퓨터에 앉힌 동료**. 폴더를 통째로 주고 "이 안에서 일해"라고 위임한다. 파일 중심, 결과물 중심. 이름에 Code가 있지만 코드 전용이 아니다 — 문서 40개 일괄 정리 같은 일에 강하다.
- 비유: Desktop은 **회의실에서 같이 일하기**, Code는 **자리 내주고 위임하기**.

**S3-2. 3초 판별법과 매트릭스**
- 3초 질문: **"결과물이 '대화'인가 '파일들'인가?"** 대화·초안·검토면 Desktop, 여러 파일을 만들고 고치는 반복 작업이면 Code.

| 상황 | 도구 | 이유 |
|---|---|---|
| 아이디어 탐색, 일회성 질문, 문서 1~3개 요약 | Desktop | 왕복 대화가 빠르다 |
| 보고서 20건 → 비교표 1개 | Code | 폴더 단위 위임 |
| 매주 반복하는 정형 산출물 | Code + Skill | 절차의 자산화 |
| 사내 시스템/외부 서비스 자료 조회 | 둘 다 (Connector) | 권한이 곧 경계 |
| 결과를 화면에서 보며 다듬기 | Desktop | 즉시 피드백 |

- 비개발자 안심 포인트: Code도 자연어로 쓴다. 명령어 암기 불필요. 권한을 물어보면 확인하고 승인하는 습관만 있으면 된다.

**S3-3. 라이브 데모: 같은 과제, 두 도구**
- 과제: 샘플 문서 묶음(가상 스타트업 자료 등)으로 비교표 만들기.
- Desktop: 파일을 끌어다 넣고 대화로. → 장점: 즉각적. 한계: 문서가 30개면?
- Code: 폴더를 주고 "전부 읽고 비교표 파일로 저장해줘". → 파일이 **생겨나는** 순간을 보여준다. 이 장면이 비개발자에게 가장 임팩트가 크다.

**S3-4. 인트로 제작기 (떡밥 회수, 이 세션의 하이라이트)**
- Before/After 화면 + 실제 사용한 프롬프트 공개.
- 핵심 고백: "저는 이걸 만들기 위해 three.js(3D 프론트엔드 기술)를 배우지 않았습니다. 제가 배운 것은 **장면, 연출, 카메라 모션, 스크롤 이벤트** — 즉 감독의 언어였습니다."
- 프롬프트의 변화 과정을 보여준다: "멋있게 만들어줘"(실패) → "카메라가 아래에서 위로 도롤리하며, 스크롤에 따라 장면이 전환되고…"(성공). **구현은 AI의 몫, 연출은 나의 몫.**
- 일반화: 보험 실무자가 배워야 할 것은 파이썬이 아니라, 자기 업무의 "연출 언어" — 좋은 심사보고서의 구조, 좋은 비교표의 기준, 좋은 요약의 조건. **그것을 문서로 쓰면 그게 Skill이다.** (4교시로 직결되는 다리)

**한 줄 요약**: "대화면 Desktop, 파일 작업이면 Code. 그리고 배워야 할 것은 구현 기술이 아니라 내 업무의 연출 언어다."

## 5. 4교시 (16:30–17:30) — 실습: Claude Desktop에서 Skill 만들기 + Custom Connector 연결

### 목표 문장
> "전원이 자기 업무 Skill 1개를 만들어 실행해보고, Connector 1개를 연결해본 상태로 퇴근한다."

### 분단위 흐름

| 시간 | 블록 | 내용 |
|---|---|---|
| 16:30–16:35 | 세팅 확인 | 로그인·화면 확인, 낙오자 조교 지원 |
| 16:35–17:00 | 실습 1 | 내 업무 Skill 만들기 |
| 17:00–17:20 | 실습 2 | Custom Connector 연결 |
| 17:20–17:30 | 마무리 | 오늘의 4문장 회수 + Next Action |

### 실습 1: 내 업무 Skill 만들기 (25분)

공통 예제로 먼저 시연(5분) → 개인 적용(20분). 공통 예제: **"경영진 보고용 1페이지 메모" Skill** (기존 기획안의 템플릿 재사용).

참가자 진행 단계:
1. 최근 한 달간 2번 이상 반복한 업무 하나를 고른다.
2. 종이(또는 메모장)에 3가지만 적는다: ① 언제 이 일을 하는가 ② 일하는 순서 ③ 결과물이 갖춰야 할 형식.
3. Claude에게 그대로 붙여넣고 요청한다: "이걸 재사용 가능한 Skill(업무 지침서)로 만들어줘. 빠진 정보가 있으면 질문 5개까지만 해줘." (기존 기획안 8장의 데모 프롬프트 재사용)
4. Claude의 질문에 답하며 지침을 완성한다.
5. Claude Desktop에 Skill로 등록한다. ⚠️ 등록 UI 경로는 버전에 따라 바뀌므로 **강의 전날 실제 화면을 캡처해서 슬라이드에 반영** (아래 체크리스트).
6. 샘플 입력으로 실행해보고, 아쉬운 점 1가지를 지침에 추가한다 — "Skill은 한 번에 완성되지 않고 기른다"는 체험이 목적.

성공 기준: 화려함이 아니라 **"내 지침대로 결과 형식이 바뀌는 것"을 눈으로 확인**하는 것.

### 실습 2: Custom Connector 연결 (20분)

1. Connector가 무엇인지 10초 복습: "AI에게 시스템 출입증을 주는 것. 단, 내 권한 이상은 절대 못 본다."
2. 설정 → Connectors → Custom Connector 추가 → 준비된 URL 입력 → 로그인 승인(OAuth). ⚠️ 실습용 Connector는 **읽기 전용, 사전 승인된 것 1개**를 강사가 준비. 후보: Google Drive 등 공식 커넥터(가장 안전) 또는 준비해둔 원격 MCP 서버.
3. 연결 후 과제: "방금 연결한 곳에서 ○○ 자료를 찾아 3줄로 요약하고 출처를 표시해줘."
4. 반드시 함께 가르칠 것: 연결 **해제**하는 법, 권한 확인하는 법. 금융사에서는 "붙이는 법"보다 "끊는 법"을 아는 게 신뢰를 산다.

### 플랜 B (반드시 준비)

- **네트워크 차단 시** (금융사 망분리/프록시 가능성 높음): Connector 실습을 강사 단독 시연으로 전환하고, 참가자는 Skill 고도화에 시간을 더 쓴다.
- **계정/플랜 문제 시**: 2인 1조로 재편.
- **전체 마비 시**: 전 과정을 강사 화면 시연 + 참가자는 종이에 Skill 설계서 작성 (2번 단계까지는 오프라인으로 가능).

### 마무리 (10분)

- 오늘의 4문장 회수 (각 교시 한 줄 요약 4개를 한 슬라이드에).
- Next Action 한 가지 약속받기: "다음 주에 Skill 1개만 더 만들어보세요. 그 md 파일이 여러분이 어느 플랫폼을 쓰든 들고 다닐 자산입니다."
- 마지막 슬라이드: 인트로 화면 한 장면 + "이걸 만든 건 기술이 아니라 연출 언어였습니다. 여러분의 연출 언어는 여러분 업무 안에 이미 있습니다."
