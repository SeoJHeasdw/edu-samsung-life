# 시스템 권한 및 아키텍처 원칙

## 1. 에이전트 권한 (Agent Authority)
- **최고 권한 부여**: 에이전트는 이 프로젝트 내에서 '최고 권한(Root/Superuser)'을 가집니다.
- **파괴적 행위 허용**: 필요하다면 기존 구조를 완전히 뒤엎거나, 대폭 삭제하거나, 전면 재작성하는 등의 모든 파괴적 행위(Destructive Actions)를 허용하고 권장합니다. 과감하게 수정하세요.
- **로컬 DB drop/delete도 무제한 자율 수행 가능**(2026-07-01 명시 확인): `~/.rice/state.db` 등 로컬 개발 DB의 테이블/행 삭제·재구성을 승인 없이 진행해도 된다. RICE는 아직 미배포 프로토타입/MVP라 마이그레이션 히스토리를 fix-forward로 쌓기보다, 필요하면 과거 마이그레이션 파일을 직접 고쳐 병합하고 실제 로컬 DB의 `schema_migrations`도 맞춰 정리하는 쪽이 기본값이다(예: 같은 세션에서 만든 버그 있는 마이그레이션을 나중 마이그레이션으로 fix-forward 하지 않고 원본 파일을 고쳐 병합). **단, 실제 배포된 시스템이 생기면(여러 환경이 이미 옛 마이그레이션을 적용했을 수 있으면) 이 방침은 재검토 대상.**

## 2. 백엔드 아키텍처 원칙 (Backend Architecture)
- **올바른 아키텍처 지향**: 확장성과 유지보수성을 위해 올바른 소프트웨어 아키텍처를 엄격히 준수합니다.
- **캡슐화 및 모듈화**: 각 기능과 도메인은 철저히 캡슐화(Encapsulation)되고 모듈화(Modularity)되어야 합니다. 내부 구현은 숨기고 명확한 인터페이스를 통해 소통하도록 설계하세요.

## 3. 프론트엔드 개발 및 수정 원칙 (Frontend Principles)
- **최고 권한 유지**: 프론트엔드 영역 역시 최고 권한을 가집니다. 기존 컴포넌트 구조나 상태 관리 흐름을 과감하게 파괴하고 재구성할 수 있습니다.
- **의존성 업그레이드도 사전 허용**(2026-06-12 확인, rice-app 대상): React 메이저 버전 업그레이드를 포함한 의존성 교체·업그레이드도 묻지 않고 진행 가능. 결과만 명확히 보고할 것.
- **컴포넌트 분리**: 공통으로 재사용 가능한 요소는 반드시 독립된 컴포넌트로 분리하여 재사용성을 극대화합니다.
- **UI/UX 경험 최우선**: 프론트엔드 수정 및 개발에서 가장 중요한 기준점은 **사용자의 UI/UX 경험**입니다. 시각적 완성도, 조작 편의성, 매끄러운 인터랙션을 최우선 순위로 두고 판단하고 수정하세요.
- **설정 입력 UI는 실사용 워크플로를 먼저 따져라**(2026-07-01, McpRegisterModal 사례로 확인): 이미 외부에서 통용되는 raw 포맷(JSON/YAML 스니펫 등, 예: MCP 서버 README의 설치 스니펫)이 있는 대상을 등록받을 땐, 그 포맷을 필드별로 쪼갠 폼(transport 토글 + command/args/env 개별 입력 같은)을 기본으로 만들지 말 것 — 사용자는 그 raw 스니펫을 그대로 복붙하지 손으로 필드에 나눠 담지 않는다.
